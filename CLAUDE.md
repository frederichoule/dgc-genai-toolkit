# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Configuration

- **Language**: TypeScript
- **Package Manager**: pnpm
- **Framework**: SvelteKit 2 + Svelte 5 (runes mode enforced via `svelte.config.js`)
- **Styling**: TailwindCSS v4 (via `@tailwindcss/vite`)
- **Database**: Drizzle ORM against Cloudflare D1 (SQLite dialect)
- **i18n**: Paraglide (`$lib/paraglide`), locales `en` / `fr`
- **Deploy target**: Cloudflare Workers via `@sveltejs/adapter-cloudflare`

## Commands

```bash
pnpm dev           # Vite dev server
pnpm build         # wrangler types --check && vite build
pnpm preview       # Run built worker via wrangler on :4173
pnpm check         # wrangler types + svelte-kit sync + svelte-check (typecheck)
pnpm lint          # prettier --check + eslint
pnpm format        # prettier --write
pnpm gen           # Regenerate Cloudflare env types (wrangler types)

pnpm db:push       # Push schema to D1 (no migration file)
pnpm db:generate   # Generate migration from schema changes
pnpm db:migrate    # Apply migrations
pnpm db:studio     # Drizzle Studio UI
```

There is no test runner configured. Treat `pnpm check` + `pnpm lint` as the verification gate.

`drizzle.config.ts` requires `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_DATABASE_ID`, and `CLOUDFLARE_D1_TOKEN` env vars for any `db:*` command that talks to D1.

## Architecture

**Cloudflare-first runtime.** The app deploys as a single Cloudflare Worker. `src/app.d.ts` exposes `App.Platform` with `env: Env` (generated by `wrangler types` — run `pnpm gen` after editing `wrangler.jsonc`). Server code accesses D1 via `event.platform.env.<BINDING>` and must call `getDb(d1)` from `src/lib/server/db/index.ts` — the Drizzle client is request-scoped, not a module singleton, because D1 bindings only exist on the request platform object.

**Schema lives in `src/lib/server/db/schema.ts`** and is the single source of truth for Drizzle. `drizzle.config.ts` is included in the TS project via a custom `svelte.config.js` typescript override, so it typechecks against the app.

**Paraglide drives routing for i18n.** `src/hooks.ts` uses `deLocalizeUrl` in `reroute` so SvelteKit's router sees de-localized pathnames — meaning route files under `src/routes/` do **not** contain a locale segment, and `localizeHref(...)` is used on the client to produce per-locale links. `src/hooks.server.ts` wraps `resolve` with `paraglideMiddleware` and injects `%paraglide.lang%` / `%paraglide.dir%` into `app.html` via `transformPageChunk`. Messages live in `messages/{locale}.json` (configured by `project.inlang/settings.json`); the compiled runtime is emitted to `src/lib/paraglide/` — treat that directory as generated.

**UI system.** See `UI.md` for the design system: theme tokens (defined in `src/routes/layout.css` under `@theme`), typography (Figtree), the `Button`/`Navbar`/`Footer` component contracts, and Svelte 5 conventions. Keep `UI.md` in sync when tokens, shared components, or patterns change. Shared components go under `src/lib/components/` (PascalCase), imported via `$lib/components/Name.svelte`.

**Svelte 5 runes only.** `svelte.config.js` forces runes mode for all non-`node_modules` files. Use `$props`, `$state`, `$derived` — no `export let`, no legacy stores for local state. Props are typed via a local `interface Props` passed to `$props()`.
